# 유니티와 객체지향 프로그래밍 (OOP)

프로그램의 모든 요소를 독립적인 '객체(Object)'로 바라보고, 이 객체들 간의 상호작용으로 로직을 구성하는 프로그래밍 패러다임.
<br> 목표 : **코드의 재사용성**을 높이고, 유지보수를 용이하게 하며, 유연한 소프트웨어 아키텍처 구축

## 캡슐화
객체의 상태(데이터)를 외부에서 무분별하게 수정하지 못하도록 하고, 허용된 방식으로만 접근하게 만드는 원칙
- 데이터 은닉 : 객체의 내부 상태를 외부에서 직접 접근하지 못하도록 숨기는 기법
- 무결성 유지 : 외부의 잘못된 접근이나 조작으로부터 객체 상태를 보호하여, 예기치 못한 버그나 사이드 이펙트를 차단

## 상속
부모 클래스가 가진 속성과 기능을 자식 클래스가 물려받아 사용하는 개념.
<br> 공통된 코드를 중복 작성할 필요가 없어짐
- 코드 재사용성 : 여러 클래스에 중복되는 데이터와 기능을 부모 클래스 (Base Class)로 추출하여 중복 코드를 제거
- 계층 구조 형성 : 'A는 B이다(IS-A)' 관계를 성립시켜, 논리적이고 체계적인 오브젝트 트리를 구축 (ex. 고블린은 몬스터이다)
- 확장성 향상 : 새로운 종류의 객체를 추가할 때, 기존 부모 클래스를 상속받아 고유한 특징만 추가 구현하면 되므로 개발 속도가 비약적으로 상승

## 다형성
호출하는 측에서는 동일한 메소드를 실행하지만, 객체의 실제 타입에 따라 각기 다른 결과가 나타나는 현상
- 의존성 감소 : 구체적인 클래스 타입(Slime, Orc 등)에 의존하지 않고, 추상적인 인터페이스(IDamageable)나 부모 타입(EnemyBase)에 의존하게 만들어 시스템의 결합도를 낮춤
- 유연한 확장 : 새로운 기능이나 객체가 추가되어도, 기존 코드를 수정할 필요없이 유연하게 대처할 수 있음

# 디자인 패턴
## 생성
- 객체 생성의 캡슐화 : 객체를 `new` 키워드나 `Instantiate()`로 무분별하게 생성하는 것을 막고, 생성 로직을 별도의 클래스나 메소드로 분리
- 유연성 및 재사용성 확보 : 시스템이 어떤 구체적인 클래스의 인스턴스를 생성해야 하는지 몰라도 되게 만들어, 코드의 결합도를 낮추고 유지보수를 용이하게 함

## 구조
- 클래스와 객체의 조합 : 객체들을 엮여서 크고 복잡한 구조를 만들면서도, 시스템의 유연성과 효율성을 잃지 않도록 돕는 패턴
- 결합도 감소 및 독립성 보장 : 서로 다른 역할을 하는 모듈 간의 결합을 느슨하게 만들어, 한쪽의 변경이 다른 쪽에 미치는 영향을 최소화 

## 행위
- 책임의 분배와 통신 : 객체들이 서로 어떻게 상호작용하고 책임을 나눌 것인지를 정의
- 결합도 최소화 (Loose Coupling) : A 객체가 B 객체에게 명령을 내리거나 상태 변화를 알릴 때, 서로의 구체적인 구현을 몰라도 되도록 설계하여 유연성을 확보
